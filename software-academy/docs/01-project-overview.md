# 01 — Обзор проекта (контекст)

Этот документ — единый источник правды о продукте. Его читают и разработчики,
и ИИ-агенты, чтобы понимать, что и зачем мы строим. Если что-то непонятно в
коде — ответ должен быть здесь.

---

## 1. Что это и зачем

**Software Academy** — учебный портал для пользователей Manufacturing Platform.
Внутри — короткие видеоуроки на 3–5 минут, привязанные к конкретным задачам
(«как составить отчёт о проверке», «как отметить часы» и т.п.).

Цели продукта:
- сократить время онбординга новых сотрудников;
- снизить нагрузку на поддержку (ответ можно найти самому);
- давать командам на производстве непрерывное обучение.

Главная цель опыта: пользователь находит нужный ответ за пару минут и
возвращается к работе. Academy должна ощущаться как продолжение промышленного
продукта, а **не** как «сайт с курсами».

## 2. Действующие лица и модель доступа

В системе три типа участников:

1. **Вендор / разработчик (мы)** — ведём код, загружаем видео на Vimeo,
   создаём уроки и категории, проставляем видимость по ролям. Контент —
   **единая библиотека для всех клиентов** (не отдельная база на каждого
   клиента). На уровне бэкенда это означает: управление контентом — наша
   зона; в API клиента таких операций нет.

2. **Администратор завода (клиент)** — компания, купившая продукт. Получает
   логин/пароль при покупке. Управляет **только людьми**: создаёт
   пользователей, назначает им роли, активирует/деактивирует, сбрасывает
   пароли. Контент не трогает.

3. **Работники** — учатся по своей роли.

### Роли

| Роль (`key`)     | Тип            | Видит уроки            | Видит раздел Users |
|------------------|----------------|------------------------|--------------------|
| `administrator`  | привилегированная | все                 | да                 |
| `manager`        | привилегированная | все *(настраивается в будущем)* | да      |
| `site_manager`   | привилегированная | все *(настраивается в будущем)* | да      |
| `inspector`      | рабочая        | помеченные под его роль | нет               |
| `welder`         | рабочая        | помеченные под его роль | нет               |
| `fitter`         | рабочая        | помеченные под его роль | нет               |

- Пользователю назначается **ровно одна** роль.
- Модель **расширяема**: новые роли добавляются без переделки логики.
- Пока `manager` и `site_manager` равны `administrator` по доступу к урокам;
  ограничения для них — задача на будущее (заложить флаг `is_privileged`).

### Правило видимости урока (важно!)

Урок виден пользователю, если выполняется **любое** из условий:

```
урок помечен как "виден всем" (is_public = true)
ИЛИ роль пользователя — привилегированная (is_privileged = true)
ИЛИ роль пользователя входит в список ролей урока
```

Примеры из ТЗ:

| Урок                     | Видят                                  |
|--------------------------|----------------------------------------|
| Create Inspection Report | Administrator, Manager, Inspector      |
| Daily Welding Report     | Administrator, Manager, Welder         |
| Login / Adding hours     | Все (`is_public = true`)               |

**Ручного назначения уроков пользователю нет.** Видимость определяется ролью.
Смена роли мгновенно перестраивает каталог пользователя — никаких доп. действий.

### Прогресс — строго личный

Чужой прогресс не видит **никто**, включая администратора. Это сознательное
ограничение: оно убирает командную аналитику из системы и упрощает архитектуру.
(Если в будущем понадобится командная статистика — это будет отдельное решение,
не в MVP.)

### Первый вход работника

Администратор создаёт пользователя и получает для него **временный пароль**.
Передаёт работнику офлайн. При первом входе работник обязан сменить пароль на
постоянный. На бэкенде это флаг `must_change_password`.

## 3. Разделы приложения (что строим в API)

| Раздел     | Назначение                          | Кому доступен        |
|------------|-------------------------------------|----------------------|
| Dashboard  | личная домашняя (continue/recommended) | все               |
| Lessons    | каталог уроков по категориям        | все (фильтр по роли) |
| Reports    | личная статистика обучения          | все                  |
| Users      | управление людьми                   | привилегированные    |
| Settings   | язык, профиль, уведомления и т.п.   | все                  |

Бэкенд должен обслуживать данные для всех этих разделов. Категории уроков —
**динамические** (CRUD на стороне вендора), потому что отражают растущую платформу.

---

## 4. Ключевые сущности (предметная область)

Коротко — подробная схема БД в `03-database-guide.md`.

- **Role** — роль (administrator, welder, …), флаг привилегированности.
- **User** — пользователь: имя, username, хеш пароля, роль, активность,
  флаг «надо сменить пароль».
- **Category** — категория уроков (может иметь подкатегорию через `parent_id`).
- **Lesson** — урок: заголовок, описание, длительность, Vimeo ID, превью,
  транскрипт, категория, флаг «виден всем».
- **LessonRole** — связь «какие роли видят этот урок» (многие-ко-многим).
- **LessonProgress** — личный прогресс пользователя по уроку: статус,
  процент просмотра, последняя позиция, временные метки.

## 5. Что НЕ входит в MVP (future)

ИИ-ассистент, обучающий аватар, квизы, сертификаты, закладки, избранное,
заметки, learning paths, уведомления, поиск по транскрипту, офлайн-режим,
скачиваемые вложения к урокам, командная аналитика, тёмная тема.

Эти пункты держим в голове при проектировании (чтобы не мешать будущему
расширению), но в коде MVP не реализуем.

---

## 6. Принципы

- Бизнес-логику держим **на бэкенде**. Фронтенд — презентационный.
- Строгая типизация (Python type hints + Pydantic, на фронте — strict TS).
- Никакого дублирования логики; переиспользуемые компоненты.
- Масштабируемая схема БД (закладываем расширение ролей и категорий).
- i18n-ready: язык по умолчанию — English.
- Доступность (на фронте — WCAG 2.1 AA).
