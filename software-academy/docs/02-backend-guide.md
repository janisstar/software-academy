# 02 — Инструкция бэкенд-разработчика (Python)

Этот документ объясняет, **как устроен бэкенд** и **как с ним работать**.
Написано для junior — без пропусков «само собой понятно».

---

## 1. Стек и зачем каждый элемент

| Инструмент          | Зачем он нужен (простыми словами)                                   |
|---------------------|---------------------------------------------------------------------|
| **FastAPI**         | Веб-фреймворк. Описываешь функцию-эндпоинт — получаешь REST API.     |
| **Uvicorn**         | Сервер, который запускает FastAPI-приложение.                       |
| **Pydantic v2**     | Описывает «форму» данных. Проверяет входящие запросы, формирует ответы. |
| **SQLAlchemy 2.0**  | ORM — позволяет работать с таблицами БД как с Python-объектами.     |
| **Alembic**         | Миграции — версии схемы БД (как git, но для структуры таблиц).      |
| **psycopg**         | Драйвер, через который Python общается с PostgreSQL.                |
| **pydantic-settings** | Читает настройки из `.env` (пароли, адрес БД и т.п.).             |
| **PyJWT**           | Создаёт и проверяет токены для авторизации.                         |
| **passlib / bcrypt**| Хеширует пароли (никогда не храним пароль в открытом виде).         |

### Почему FastAPI, а не Django или Flask?

- **Swagger из коробки.** Ты описываешь типы — FastAPI сам строит интерактивную
  документацию на `/docs`. Не надо писать её вручную. Это прямо твоё требование.
- **Валидация бесплатно.** Если клиент пришлёт кривые данные — FastAPI вернёт
  понятную ошибку 422 сам, до твоего кода.
- **Подсказки в редакторе.** Из-за типов IDE подсказывает поля — junior меньше ошибается.

> Запомни: каждый эндпоинт = обычная Python-функция с типами. Магии нет.

## 2. Структура папок (feature-ready, по слоям)

```
backend/
├── app/
│   ├── main.py            # точка входа: создаёт FastAPI-приложение
│   ├── core/
│   │   ├── config.py      # настройки из .env (адрес БД, секрет JWT)
│   │   ├── database.py    # подключение к БД, сессии
│   │   └── security.py    # хеш паролей, создание/проверка JWT
│   ├── models/            # таблицы БД (SQLAlchemy): user.py, lesson.py, ...
│   ├── schemas/           # формы запросов/ответов (Pydantic): user.py, ...
│   ├── services/          # бизнес-логика: что делать с данными
│   └── api/
│       └── v1/            # роуты (эндпоинты): auth.py, users.py, lessons.py, ...
├── alembic/               # миграции БД
├── tests/                 # тесты (на будущее)
├── requirements.txt       # список зависимостей
├── .env.example           # шаблон настроек (без секретов)
├── Dockerfile
└── docker-compose.yml
```

### Что в каком слое лежит (это важно понять)

Один запрос проходит через слои сверху вниз:

```
HTTP-запрос
   │
   ▼
api/v1/  ──── роут: принимает запрос, проверяет права, зовёт сервис
   │
   ▼
schemas/ ──── проверяет/формирует данные (Pydantic), что пришло и что вернём
   │
   ▼
services/ ─── бизнес-логика: «можно ли этому юзеру видеть этот урок?»
   │
   ▼
models/  ──── таблицы БД (SQLAlchemy), запрос к PostgreSQL
   │
   ▼
PostgreSQL
```

**Правило:** роут (`api/`) — тонкий, он только координирует. Вся логика «как
считать, кому что можно» — в `services/`. Так логика не дублируется и легко
тестируется.

| Слой        | Можно ли тут...                          | Пример                          |
|-------------|------------------------------------------|---------------------------------|
| `api/`      | принять запрос, проверить роль, вернуть   | `@router.get("/lessons")`       |
| `schemas/`  | описать поля запроса/ответа               | `class LessonOut(BaseModel)`    |
| `services/` | принять решения, посчитать                | `get_lessons_visible_to(user)`  |
| `models/`   | описать таблицу                           | `class Lesson(Base)`            |

## 3. Соглашения (договорённости в команде)

- **Язык кода — английский** (имена переменных, функций, таблиц). Комментарии
  можно по-русски — для понимания junior.
- **Типы обязательны.** У каждой функции — типы аргументов и возврата.
- **Имена файлов в моделях/схемах — в единственном числе:** `user.py`, `lesson.py`.
- **Таблицы в БД — во множественном числе и snake_case:** `users`, `lesson_progress`.
- **Эндпоинты версионируем:** все под `/api/v1/...`. Появится несовместимая
  версия — будет `/api/v2/`, старая не сломается.
- **Секреты только в `.env`**, который в git не попадает (см. `.gitignore`).
  В репозитории лежит `.env.example` — шаблон без реальных значений.

## 4. Рецепт: как добавить новый эндпоинт (по шагам)

Допустим, нужен эндпоинт «получить список категорий». Делаем так:

1. **Схема ответа** — `app/schemas/category.py`:
   опиши, какие поля вернём (Pydantic-модель `CategoryOut`).

2. **Логика** — `app/services/category_service.py`:
   функция `list_categories(db)`, которая берёт данные из БД.

3. **Роут** — `app/api/v1/categories.py`:
   ```python
   @router.get("/categories", response_model=list[CategoryOut])
   def get_categories(db: Session = Depends(get_db)):
       return category_service.list_categories(db)
   ```

4. **Подключи роут** в `app/main.py` (или в общем роутере `api/v1`):
   ```python
   app.include_router(categories.router, prefix="/api/v1", tags=["categories"])
   ```

5. **Проверь в Swagger:** запусти сервер, открой `http://localhost:8000/docs`,
   найди новый эндпоинт, нажми «Try it out».

Эта же схема (schema → service → route → include → проверить в Swagger)
повторяется для всего. Разберёшь её один раз — дальше пойдёт быстро.

## 5. Авторизация — как это будет работать (обзор)

Подробно реализуем на отдельном шаге roadmap. Кратко логика:

1. Пользователь шлёт `POST /api/v1/auth/login` с username + password.
2. Бэкенд находит юзера, сверяет пароль с хешем (`security.py`).
3. Если ок — выдаёт **JWT-токен** (строку), внутри которой зашит id и роль.
4. Дальше фронт прикладывает токен в заголовке `Authorization: Bearer <токен>`
   к каждому запросу.
5. Защищённые эндпоинты через `Depends(get_current_user)` достают юзера из токена.

Если у юзера `must_change_password = true`, после логина фронт ведёт его на
смену пароля, и пока он не сменит — доступ к остальному ограничен.

## 6. Полезные команды (запомни)

```bash
# создать виртуальное окружение (изолированные зависимости проекта)
python -m venv .venv
source .venv/bin/activate        # Windows: .venv\Scripts\activate

# установить зависимости
pip install -r requirements.txt

# запустить сервер для разработки (--reload = перезапуск при изменениях)
uvicorn app.main:app --reload

# открыть документацию API
# http://localhost:8000/docs   (Swagger)
# http://localhost:8000/redoc  (альтернативный вид)
```

Запуск через Docker и работа с миграциями — в `04-server-and-docker.md` и
`03-database-guide.md`.
