# 06 — Конвенции API (единый стиль с платформой)

Academy строится в одном стиле с **Welding Log API** (экосистема SevenHeaven),
но как **отдельное приложение** со своей БД и своим входом. Этот документ —
источник правды по форме API и авторизации. При расхождении с другими доками
приоритет здесь.

---

## 1. Базовые конвенции

- **Префикс без версии:** все эндпоинты под `/api/...` (как в Welding Log,
  а не `/api/v1/...`).
- **Слеш на конце пути:** `/api/login/`, `/api/user/`, `/api/users/`.
- **Единственное число — одна запись, множественное — список:**
  `/api/user/` (POST/PATCH/DELETE), `/api/users/` (список),
  `/api/users/paginated`, `/api/user/{username}` (по имени).
- **Группировка по тегам** в Swagger: Health, Authentication, Users, Companies,
  Categories, Lessons, Progress, Reports, Settings.
- **Ошибка валидации — 422** в формате FastAPI (`{"detail": [...]}`) — как у Welding Log.
- **Создание — 201**, успех — 200, нет прав — 403, не найдено — 404.

---

## 2. Роли (один пользователь — одна роль)

Ключи ролей совпадают с платформой Welding Log (один человек — один ключ).
**Убраны:** `client`. **Сохранён:** `master` (платформенная роль вендора).

| key         | Название       | Уровень   | Видит уроки | Управляет людьми          | Управляет контентом |
|-------------|----------------|-----------|-------------|---------------------------|---------------------|
| `master`    | Master         | платформа | (все, для управления) | всеми компаниями  | **да** (только он)  |
| `admin`     | Administrator  | компания  | все         | своей компании            | нет                 |
| `manager`   | Manager        | компания  | все         | своей компании            | нет                 |
| `site`      | Site Supervisor| компания  | все         | своей компании            | нет                 |
| `inspector` | Inspector      | компания  | свои        | нет                       | нет                 |
| `user`      | Welder         | компания  | свои        | нет                       | нет                 |
| `fitter`    | Fitter         | компания  | свои        | нет                       | нет                 |

- **master** — наш вход как вендора: наполняет каталог (категории, уроки,
  Vimeo ID, видимость по ролям) и помогает любой компании с доступом/данными.
  Уроки не смотрит, прогресса не имеет.
- `admin / manager / site` — `is_privileged = true`: видят весь каталог и
  управляют пользователями **своей** компании.
- `inspector / user / fitter` — рабочие: видят только уроки своей роли.

> Историческое соответствие: `administrator → admin`, `site_manager → site`,
> `welder → user`. `manager / fitter / inspector` — без изменений.

## 3. Компании (мульти-тенант в одной БД)

Одно развёртывание обслуживает все компании-клиенты. Контент (категории/уроки) —
**общий для всех**; пользователи **привязаны к компании** через `company_id`.

- `master` видит все компании; `admin/manager/site` — только свою.
- Служебная **«компания-платформа»** (`id = 1`, напр. *SevenHeaven*) — к ней
  привязаны master-пользователи. Так у каждого пользователя всегда есть
  `company_id`, без пустых значений.

Поля компании (простой вариант): `id`, `name`, `businessid` (необяз., рег.номер/VAT),
`email` (необяз.), `is_locked`, `created_at`. **Убрано** из Welding Log:
`privileges` (master/retailer), `mastercompanyid`.

## 4. Авторизация — серверные сессии (безопасный вариант)

Выбран самый надёжный и современный вариант. Пути узнаваемые (как у Welding Log),
но передача данных — безопаснее, чем через query-параметры.

**Вход `POST /api/login/`**
- Принимает `{ "un": "...", "pw": "..." }` **в теле запроса** (не в URL).
- Сервер сверяет пароль с хешем (**Argon2id**), создаёт запись в таблице
  `sessions` и возвращает идентификатор сессии в **httpOnly + Secure + SameSite cookie**.
- Cookie httpOnly недоступна JavaScript → защита от XSS-кражи сессии.

**Запросы после входа**
- Браузер сам присылает cookie с сессией. Бэкенд по ней находит пользователя в БД.
- В Swagger это работает прозрачно: вошёл через `/api/login/` — последующие
  «Try it out» уже идут с cookie.

**Выход `POST /api/logout/`**
- Удаляет запись сессии из БД (доступ пропадает **мгновенно**) и стирает cookie.

**Таблица `sessions`:** `id` (случайный токен), `user_id`, `created_at`,
`expires_at`, `last_seen_at`, `ip`, `user_agent`. Срок жизни — настройка
(`SESSION_TTL`): сейчас длинный, позже можно ~1 час.

> Почему так, а не query-параметры как в Welding Log: пароли и сессии не должны
> попадать в URL (логи, история браузера). Это сознательное отклонение в пользу
> безопасности — пути и смысл эндпоинтов при этом сохранены.

Зависимости (добавим на Шаге авторизации): `argon2-cffi` для хеша паролей.

## 5. Форма пользователя в ответе (как у Welding Log)

Храним **одну роль** (`role_id`), но в ответе отдаём привычный `privileges`-объект,
собранный из неё. Так форма совпадает с платформой, а хранение остаётся простым.

```jsonc
{
  "id": 12,
  "un": "j.salmela",
  "name": "Jenna Salmela",
  "email": "jenna@acme.fi",     // необязательное
  "companyid": 3,
  "privileges": {               // собрано из одной роли user=1
    "master": 0, "admin": 0, "manager": 0, "site": 0,
    "user": 1, "fitter": 0, "inspector": 0,
    "locked": 0                 // locked = 1, если пользователь заблокирован
  }
}
```

Правило сборки: ключ текущей роли = 1, остальные = 0; `locked` = 1, если
`is_locked`. **Убрано** относительно Welding Log: `phone`, `languageid`,
`client`, `terms_and_usage`.

## 6. GDPR (закладываем сразу)

- Храним минимум персональных данных (имя, username, опц. email); пароль — только хешем.
- `DELETE /api/user/` — реальное удаление/обезличивание пользователя и его
  прогресса («право на забвение»).
- `POST /api/user/lock/` — временная блокировка вместо удаления (данные сохраняются).
- Возможность выгрузить свои данные по запросу.

---

## 7. Что берём из Welding Log, а что нет

| Welding Log | В Academy |
|---|---|
| `/api/login/`, `/api/logout/` | да |
| `/api/user/` POST/PATCH/DELETE, `/api/users/`, `/api/user/{username}` | да |
| `/api/user/lock/` | да |
| password-reset через код (15 мин) + confirm, generate password | да |
| `sessionID` в query | **нет** → httpOnly-cookie |
| `un/pw` в query | **нет** → в теле запроса |
| `privileges`-объект в ответе | да (но собран из одной роли) |
| `client`, лицензии, Telegram SSO, `terms_and_usage` | **нет** |
| `phone`, `languageid`, `mastercompanyid` | **нет** |
| WebSocket-стримы | не в MVP |
